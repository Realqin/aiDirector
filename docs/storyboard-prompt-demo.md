# AI 分镜 · 阶段性提示词 Demo

本文提供一套可端到端跑通的**示例主题**与各节点**提示词范本**，便于在「提示词管理」中新建模板并绑定到对应节点，或与前端内置 `defaultPromptMap` 对照使用。

**使用方式建议**

- 在系统中为每个阶段建一条提示词模板，**描述**字段填入下面对应全文；分镜运行页自定义节点里把该模板绑定到同名节点。
- 自定义输入框里可追加本 Demo 的「统一主题」或客户的具体素材，无需改模板。

---

## 统一 Demo 主题（建议在首节点自定义区粘贴）

```text
【Demo 主题】60 秒治愈系短视频《深夜便利店》
核心：夜班店员小默与一位冒雨进店的老顾客之间，用一杯热饮完成无声陪伴。
调性：暖色、克制、少对白；结尾一句旁白点题。
禁止：恐怖、狗血反转、品牌硬广。
```

## 阶段 1：故事描述

**目标**：产出可供拆场景的叙事文本（纯文本即可）。

**提示词 Demo：**

```text
你负责短视频叙事打底。根据用户给出的主题与约束，用简洁中文写一段完整故事梗概（约 300～500 字）。

必须包含：
- 主角是谁、当下处境与动机；
- 情绪曲线（起承转合一句话各一句）；
- 适合影像化的 2～4 个关键瞬间（不必编号，自然写在文中）。

风格：口语化叙事，避免分镜术语与镜头号；不要输出 JSON 或列表标题。
```

---

## 阶段 2：场景分解

**目标**：与前端「场景分解」卡片一致，模型须输出**严格 JSON**（无 Markdown 代码块）。

**提示词 Demo：**

```text
你是短视频编剧助理。请根据【上一节点给出的故事/素材】将内容拆为若干场景。

硬性要求：
1. 只输出一个 JSON 对象，不要用 ``` 包裹，不要任何解释性前后文。
2. 结构固定如下（字段键名必须英文小写，与示例一致）：

{"scenes":[{"title":"场景标题","location":"地点","time":"时间或时段","characters":"出场人物","conflict":"冲突点或戏剧张力","visualFocus":"本场景视觉重点（光影/道具/构图关键词）"}]}

3. 场景数量 3～6 个为宜；每个字段用简洁中文填写，勿留空对象。
4. 若用户要求整体改写，在保留 scenes 数组结构的前提下替换各字段文案。
```

**输出示例（结构参考）：**

```json
{"scenes":[{"title":"雨夜店外","location":"便利店门口与城市街道","time":"深夜","characters":"小默（店员）","conflict":"雨声与霓虹反差，等人少","visualFocus":"玻璃上的雨痕、冷蓝霓虹"}]}
```

## 阶段 3：场景评审

**目标**：对「场景分解」产出的 JSON（或合并后的正文）做结构化评审，输出**严格 JSON**。

**提示词 Demo：**

```text
你是对短视频分镜负责的场景编辑。上游会提供待评审的场景内容（可能是 JSON 字符串或叙述文本）。

请对比「原内容」与一条明确的期望标准（节奏是否清晰、场景是否可拍、人物动机是否连贯），输出严格 JSON（不要 markdown 代码块），格式如下：

{"评审结论":"整体评价与是否建议进入画面拆解（可多句）","问题点":["具体问题1","问题点2"],"原内容":"将待评审内容原文粘贴于此（若上文已给全文可复述关键段）","修改后":"修改润色后的完整正文（若原内容为 JSON，修改后仍为合法 JSON 字符串，结构与键名与场景分解阶段一致）"}

要求：
- 问题点用 JSON 数组，每条一句；
- 原内容、修改后均为字符串；
- 修改后应保留原意并修正问题；若原内容为 scenes JSON，修改后必须仍是可解析的同一结构。
```

---

## 阶段 4：画面分解

**目标**：按场景拆「画面」层级，输出**严格 JSON**，供前端卡片展示。

**提示词 Demo：**

```text
你是短视频分镜师。根据上游「场景」与评审结论，整理为按场景、按画面的结构化分镜。

只输出 JSON，不要用 ``` 包裹。格式如下：

{"scenes":[{"title":"场景标题（可与上游一致或略精简）","frames":[{"description":"画面1：景别+主体+动作+光影+情绪，一句话"},{"description":"画面2：..."}]}]}

规则：
- 每场景 3～6 个画面为宜；description 用中文，可直接写画面内容，也可用「画面N：」前缀。
- 若用户要求整体改写，在保留 scenes 与 frames 数组结构的前提下替换文案。
- 不要输出 JSON 以外的文字。
```

---

## 阶段 5：画面评审

**目标**：对画面级描述做专业点评（本节点前端未强制 JSON，可用结构化段落）。

**提示词 Demo：**

```text
你从构图、光影、连贯性、可执行性（是否方便实拍或 AI 出图）评审下列分镜画面描述。

输出结构（用中文，可用 Markdown 标题）：
## 总评
（2～4 句）

## 优点
- 条目列表

## 风险与改进
- 条目列表（指出跳跃、信息缺失、景别混乱等）

## 优化后的画面说明（按场景分组）
对每个场景，用「场景名」下重新列画面 1、画面 2… 的推荐描述，便于直接替换上游 JSON 中的 frames.description。
```

## 流水线自检清单（演示时快速核对）

| 阶段       | 前端主要形态     | 模型输出要点                 |
|------------|------------------|------------------------------|
| 故事描述   | 纯文本输出       | 无 JSON，一篇梗概            |
| 场景分解   | 场景卡片 + JSON  | 顶层 `scenes` 数组           |
| 场景评审   | 对比视图 + JSON  | 含 评审结论/问题点/原/改后   |
| 画面分解   | 画面卡片 + JSON  | `scenes[].frames[]`          |
| 画面评审   | 纯文本输出       | 分段标题 + 列表 + 优化稿     |

将本文件与项目内 `StoryboardRunView.vue` 中 `defaultPromptMap` 对照，即可逐步替换为更贴合业务的版本。
